# what is IOC

IoC means **Inversion of Control**. 

IoC is the idea that Spring controls object creation and dependency management instead of us doing it manually.

This helps make the code loosely coupled, easier to test, and easier to maintain. 

- It is easier to test because we can inject mock objects in unit tests.
- It is easier to maintain because if we replace one implementation with another, we do not need to change all the classes that depend on it. Spring can inject the new implementation for us.

DI is a way to achieve IoC, where Spring injects the needed dependencies into an object.

- Typical DI implementations include constructor injection, setter injection, and field injection.
- For example, we can annotate a service class with @Service, a repository with @Repository, and a controller with @RestController. These annotations tell Spring to create and manage these classes as beans.
- Then, if StudentController needs StudentService, we do not write new StudentService(). Instead, we define a constructor parameter, and Spring automatically finds the StudentService bean and injects it.

So the control of object creation is moved from our code to the Spring container.

# what is AOP

AOP means separating common logic  known as cross-cutting concerns from the main business logic.

For example, logging, security checks, transactions, or exception handling can be handled in one place instead of being repeated in many methods.

With AOP, we can define this common logic once and apply it automatically before, after, or around certain methods.

We can use `@Aspect` on a class to mark it as an aspect. This means the class is used to handle common cross-cutting logic, such as logging, security, or transaction-related logic. Inside the aspect class, we can define methods that run before, after, or around certain business methods. We can also use `@Pointcut` to define where the AOP logic should be applied. For example, a pointcut can match all methods in a service package.



# what is Aspect, joinpoint, pointcut, advice, target

An aspect is a class used to handle common cross-cutting logic, such as logging, security, or transaction-related logic. Inside the aspect class, we can define methods that run before, after, or around certain business methods. 

- We can use `@Aspect` on a class to mark it as an aspect. 



A join point is a point during program execution where AOP logic can be applied.

- In Spring AOP, a join point usually means a method execution.
- For example, when `userService.createUser()` is called, that method call is a join point.



A pointcut decides where the AOP logic should be applied.

- We can use @Pointcut to define a reusable pointcut expression. 
- Then we can use it with advice annotations like @Before, @After, or @Around.



Advice means what AOP logic should be executed.

- For example, the advice can be logging, permission checking, transaction handling, performance monitoring, or idempotency checking.
- In Spring AOP, common advice annotations include @Before, @After, @AfterReturning, @AfterThrowing, and @Around.
- @Before means the logic runs before the target method.
- @After means the logic runs after the target method, no matter whether it succeeds or throws an exception.
- @AfterReturning means the logic runs only after the target method successfully returns.
- @AfterThrowing means the logic runs only when the target method throws an exception.
- @Around  runs logic before and after the target method, and it can even decide whether to continue calling the target method.



Target means the actual object that is being advised by AOP.

- For example, if we define an AOP logging logic for all methods in the service layer, then the service-layer beans, such as UserService or OrderService, are target objects.
- The methods inside those classes are target methods.





# ApplicationContext vs BeanFactory

`BeanFactory` and `ApplicationContext` are both Spring IoC containers.

`BeanFactory` is the basic container. It can create beans, manage beans, and inject dependencies.

`ApplicationContext` is the more advanced container. It includes everything from `BeanFactory`, and also provides extra features like event publishing, internationalization, annotation support, and integration with Spring Boot.

In real Spring Boot projects, we usually use `ApplicationContext`, not `BeanFactory` directly.

By default, `BeanFactory` usually creates beans lazily, which means beans are created when they are requested.

`ApplicationContext` usually creates singleton beans eagerly during application startup, so configuration problems can be found earlier.







# How is Spring MVC Working Flow 

Spring MVC works around the **DispatcherServlet**.

When the client sends a request, and the `DispatcherServlet` receives it first. Then it uses `HandlerMapping` to find the correct controller method. After that, `HandlerAdapter` calls the controller method. The controller handles the request, usually by calling the service layer. Then it returns either a view name or response data. If it is a traditional MVC application, `ViewResolver` resolves the view and returns an HTML page. If it is a REST API, Spring directly returns JSON data to the client.



# how does @Autowire work

`@Autowired` tells Spring to inject a dependency

When Spring creates a bean, it checks whether the bean needs other dependencies. If it sees `@Autowired`, Spring looks inside the IoC container for a matching bean. Usually, it matches by type. Then Spring injects that bean into the current class. If there are multiple beans of the same type, Spring needs extra information, such as `@Qualifier` or `@Primary`. If there is no matching bean, Spring will throw an error unless the dependency is optional.





# @Autowired and @Qualifier

`@Qualifier` is usually used together with `@Autowired`. 

`@Autowired` tells Spring to inject a dependency, and `@Qualifier` tells Spring which specific bean to inject when there are multiple beans of the same type.

But in constructor injection, if there is only one constructor, `@Autowired` can be omitted, and `@Qualifier` can still be used on the constructor parameter. Constructor injection is generally recommended because it makes dependencies explicit, supports immutability, and makes testing easier.

For setter injection and field injection, `@Autowired` cannot be omitted. 

- Field injection is not recommended because it hides dependencies, makes the class harder to test, and makes it harder to pass mock objects in unit tests. 
- Setter injection can be useful for optional dependencies because we can choose whether to provide the dependency or not.



```java
@Component
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(@Qualifier("PaypalPaymentService") PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
```





# 2 use cases for @PostConstruct and @PreDestroy

`@PostConstruct` runs **after the bean is created and dependencies are injected**.

**Use case 1: load business configuration**

`@PostConstruct` can be used to load business configuration after dependency injection is finished. This avoids querying the database or remote config center every time the business logic runs. But `@PostConstruct` only runs once during bean initialization, so if the configuration changes later, it will not refresh automatically.

**Use case 2: initialize cache**

`@PostConstruct` can be used to initialize cache when the application starts. We can load frequently accessed and rarely changed data into memory. Then the business logic can read from local cache instead of querying the database every time. This can reduce database pressure and improve performance. But we need to consider cache refresh and data consistency across multiple service instances.



`@PreDestroy` runs **before the bean is destroyed**.

**Use case 1:  close the thread pool**

- `@PreDestroy` can be used to close a custom thread pool before the Spring container shuts down. This helps prevent background threads from continuing to run after the application is stopping. It can also avoid resource leaks and make sure the JVM can exit normally. 

- When closing a thread pool, we should understand the difference between `shutdown()` and `shutdownNow()`. `shutdown()` is a graceful shutdown. It stops accepting new tasks but tries to finish existing tasks. `shutdownNow()` is more aggressive because it tries to interrupt running tasks. So in most cases, we should prefer graceful shutdown.

**Use case 2:  release file resources**

- `@PreDestroy` can be used to release file resources before the Spring bean is destroyed. This helps avoid file handle leaks and prevents temporary files from occupying disk space for too long. 

- But for short-lived file streams, we should usually use `try-with-resources`. `@PreDestroy` is more suitable for cleaning up long-lived file resources.

 



`@PostConstruct` 和 `@PreDestroy` 是 Java/Spring 中用于管理 Bean 生命周期的两个注解。

1. 初始化缓存

   - `@PostConstruct` 可以在项目启动时，把一些经常访问、变化不频繁的数据提前加载到内存里，比如字典、地区、商品分类、权限菜单。这样后续业务查询时可以直接读本地缓存，减少数据库访问，提高性能。

   - 注意事项：缓存数据要考虑刷新机制和分布式多节点一致性问题。

2. 加载配置

   - 有些业务配置可能存放在数据库或配置中心，比如风控开关、支付渠道配置、订单超时时间。可以在 `@PostConstruct` 中提前加载这些配置，避免每次业务执行时都去查数据库或远程配置。（启动时先把配置读出来，后面业务执行时直接从内存拿，速度更快。）

   - 注意事项：`@PostConstruct` 只会执行一次，配置变更后不会自动刷新。



1. 关闭线程池 
   - @PreDestroy 可以在 Spring 容器关闭前，主动关闭业务中创建的线程池。 这样可以避免应用退出时还有任务线程继续运行，导致资源泄漏或进程无法正常结束。 
   - 注意事项：关闭线程池时要区分 shutdown() 和 shutdownNow()，尽量做优雅关闭。



2. 释放文件资源
   - 如果 Bean 中打开了文件流、日志文件、临时文件或本地缓存文件，可以在 `@PreDestroy` 中释放。这样可以避免文件句柄未关闭，或者临时文件长期占用磁盘空间。
   - 注意事项：文件流最好用 `try-with-resources` 管理，`@PreDestroy` 更适合清理长期持有的文件资源。
