# Why spring boots? pros and cons

We use **Spring Boot** because it makes Spring application development faster and easier.

It provides auto-configuration, embedded servers, starter dependencies, and production-ready features like Actuator.

So we can focus more on business logic instead of writing too much boilerplate configuration.



Pros

- Spring Boot is built on top of Spring, so its benefits first come from Spring core features like **IoC / DI** and **AOP**.
- With **IoC and DI**, classes do not create dependencies by themselves using `new`. Instead, Spring creates and injects dependencies for us. This makes the code more loosely coupled, easier to test with mocks, and easier to maintain when implementations change.
- With **AOP**, we can separate cross-cutting concerns from business logic. For example, logging, transaction management, security checks, and monitoring can be handled outside the main business code. This makes the business logic cleaner.
- On top of that, Spring Boot provides **auto-configuration**, **starter dependencies**, **embedded Tomcat**, and **Actuator**. These features reduce boilerplate configuration and make it faster to build production-ready applications.



Cons

- The first downside is that auto-configuration can feel like magic. If something is configured incorrectly, debugging may be harder.
- The second downside is that the application may start with many dependencies, so it can be heavier than a very simple Java application.



# How to start a spring boot project from scratch?

I would start by clarifying what kind of application we are building. If it is a web application, I would usually use IntelliJ IDEA to create a Spring Boot project. IntelliJ can generate the project structure for me, and I can choose the build tool, Java version, Spring Boot version, and required dependencies such as Spring Web, Spring Data JPA, Validation, database driver, and Actuator if needed.

After that, I would configure `application.properties` or `application.yml`, and set up different profiles for different environments, such as `dev`, `staging`, and `prod`.

After that, I would create the standard layers, like controller, service, repository, entity, and DTO. Then I would implement the APIs, add validation, global exception handling, database configuration, and tests. Finally, I would run the application locally and package it as an executable JAR for deployment.



# @Controller vs @RestController

`@Controller` is usually used for traditional web applications that return pages, while `@RestController` is usually used for REST APIs that return data directly.

`@Controller` registers a class as a Spring MVC controller and also as a Spring bean.

In a `@Controller` class, the return value of each method is treated as a view name by default. Spring passes the view name to the `ViewResolver`, and the `ViewResolver` finds and renders the corresponding page.

`@RestController` is equivalent to `@Controller` plus `@ResponseBody`.

In methods defined in a `@RestController`, if a method returns a Java object, Spring serializes it into JSON and writes it directly to the HTTP response body.



# @PathVariable vs @RequestParam

Both `@PathVariable` and `@RequestParam` are used to get data from an HTTP request, but they are used in different places.

`@PathVariable` reads values from the URL path. We use `@PathVariable` when the value identifies a specific resource.

`@RequestParam` reads values from query parameters. We use `@RequestParam` for filtering, searching, sorting, or pagination.







查询单个资源，用 `@PathVariable`，表示资源 ID

查询列表、分页、筛选，用 `@RequestParam`，表示查询条件

- `GET /orders?page=1&size=20&status=PAID`
- `@RequestParam` 默认client必须要传这些字段，可以设置`required = false` 表示非必填：

```java
@GetMapping("/orders")
public List<OrderVO> listOrders(
        @RequestParam Integer page,
        @RequestParam Integer size,
        @RequestParam(required = false) String status) {
    return orderService.listOrders(page, size, status);
}
```

语义是：查询订单列表，条件是 page、size、status





# @RequestBody vs @ResponseBody

`@RequestBody` is used to read data **from the HTTP request body**. It is usually used on controller method parameters in `POST`, `PUT`, or `PATCH` requests to read JSON data from the request body.

`@ResponseBody` is used to write data **to the HTTP response body**. It can be used on a **controller method**, and it can also be used at the **class level**. If we use `@RestController`, we usually do not need to write `@ResponseBody` manually. Because `@RestController` is equivalent to @Controller + @ResponseBody



`@RequestBody` 用在 **Controller 方法参数上**。

```java
@PostMapping("/users")
public UserVO createUser(@RequestBody CreateUserRequest request) {
    return userService.createUser(request);
}
```

DTO：

```java
public class CreateUserRequest {
    private String username;
    private String email;
    private Integer age;
}
```

这里 Spring 会把 JSON 自动转换成 `CreateUserRequest` 对象。



`@ResponseBody` 用在 **Controller 方法上**，也可以用在类上。

它的作用是：把方法返回值直接写入 HTTP 响应体。



# how to use GetMapping, PutMapping, PostMapping, DeleteMapping, RequestMapping?

`@RequestMapping` is usually used at the class level to define the base URL.

`@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping` are usually used at the method level.

- `@GetMapping` is used to get data.
- `@PostMapping` is used to create data.
- `@PutMapping` is used to fully update data.
- `@DeleteMapping` is used to delete data.



# What is Spring Actuator?

Spring Boot Actuator is a Spring Boot feature for **monitoring and managing an application**.

It provides built-in endpoints to check application health, metrics, environment, beans, logs, and other runtime information.

We need Actuator because in production, we need to know whether the application is running correctly.

Common Actuator endpoints include:

```Plain
/actuator/health
/actuator/metrics
/actuator/info
/actuator/beans
/actuator/env
/actuator/prometheus
```

In my Spring Boot project, I used Actuator to expose health status and metrics.

Then Prometheus scraped the metrics from `/actuator/prometheus`, and Grafana showed them in dashboards.

# how to achieve async in spring boot application

`@Async` 放在方法上，用来把一个方法提交到线程池中执行，让调用方不必阻塞等待方法执行完成。

使用`@Async` 前要在**Spring Boot 启动类** 或者一个 **配置类** 上加上`@EnableAsync`开启异步能力

`@Async` 本质上是基于 Spring AOP 实现的。Spring 会给被 `@Async` 标注的方法所在 Bean 创建代理对象。当调用异步方法时，代理对象会把这个方法封装成任务，提交到线程池中执行。

# how does spring validate data

Normally, we put validation annotations on DTO fields.

- Some commonly used annotations are:
  - `@NotNull` — value cannot be null 
  - `@NotBlank` — string cannot be null or empty 
  - `@Size` — validate string length or collection size 
  - `@Min` and `@Max` — validate numeric range 
  - `@Email` — validate email format 
  - `@Pattern` — validate with a regular expression 
  - `@Positive` and `@PositiveOrZero` — validate numeric values

Then we use `@Valid` in the controller method. When a request comes in, Spring validates the request body before entering the business logic. If validation fails, Spring throws a validation exception and returns an error response.

In real projects, I'll also use`@ControllerAdvice` on GlobalExceptionHandler class. This allows me to catch validation exceptions globally and return a consistent error response format.
