# What annotations have you used in Spring?

I used Spring annotations in several groups: IOC, DI, AOP, REST endpoint design, configuration, and validation.

For IOC, I used `@SpringBootApplication` to enable auto configuration and component scanning. Then I used `@Controller`, `@RestController`, `@Service`, `@Repository`, and `@Component` to register beans in different layers. For third-party objects, I used `@Configuration` and `@Bean`.

For third-party beans or beans we cannot annotate directly, I used `@Configuration` and `@Bean`.

For dependency injection, I used constructor injection most often. I also know `@Autowired`, `@Qualifier`, and `@Lazy`.

For AOP and exception handling, I used `@RestControllerAdvice`, `@ExceptionHandler`, `@Aspect`, `@Pointcut`, `@Around`, `@Before`, `@After`, `@AfterThrowing`, and `@AfterReturning`.

For REST APIs, I used `@RequestMapping`, `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@RequestParam`, `@PathVariable`, `@RequestBody`, and `@RequestHeader`.

For validation, I used annotations like `@Valid`, `@NotNull`, `@NotBlank`, and `@Size`.





# What is Enable Auto Configuration?

Enable auto configuration means Spring Boot automatically configures beans based on the dependencies we add to the project.

For example, if we add a web starter, Spring Boot can configure the embedded server and web-related components. If we add a database dependency, Spring Boot can help configure database-related beans based on the classpath and application properties.

This is part of `@SpringBootApplication`. It helps reduce manual configuration and boilerplate code.

If we do not want a specific auto configuration, we can exclude it.



# What are the advantages of Spring Boot Framework?

The main advantages of Spring Boot are auto configuration, starter packages, embedded server, externalized configuration, and production-ready features.

First, auto configuration reduces manual setup. Spring Boot can create many default beans based on the dependencies and properties.

Second, starter packages simplify dependency management. For example, we can add one starter instead of adding many small dependencies one by one.

Third, Spring Boot has embedded servers like Tomcat, so we do not need to deploy a WAR file to an external server.

Fourth, it supports properties and YAML files, so configuration is easier to manage.

Fifth, it provides Actuator, so we can expose health, metrics, and monitoring endpoints.

Also, Spring Boot is built on top of Spring, so we can still use Spring features like IOC, DI, AOP, transaction management, and Spring MVC.



# How do you define profile?

Spring Boot profiles are used for environment segregation.

Different environments usually need different configurations. For example, we may have dev, QA, UAT, staging, and production environments. Each environment can have its own database URL, Redis address, logging level, and external service configuration.

We can define profile-specific configuration files, like application-dev.yml, application-qa.yml, application-staging.yml, and application-prod.yml.

Then we activate one profile by setting `spring.profiles.active`.

The main purpose is not just to avoid hard coding. The main purpose is to separate environment-specific configurations clearly.



# Why AOP?

AOP is used to enhance functionality by breaking the original serial execution flow.

In normal business logic, the code runs step by step. With AOP, Spring can insert extra logic before, after, around, or when an exception is thrown from a target method.

The main benefit is that we can avoid duplicate code patterns. For example, we do not want to write the same logging, security check, timer, or exception logging logic in every service method.

So AOP separates cross-cutting concerns from business logic.

In Spring, we define where to apply the logic by using a pointcut, and we define when to apply it by using advice, such as around, before, after, after throwing, and after returning.

Common use cases are logging, login check, security check, performance timer, transaction-like logic, idempotency check, and exception logging.

# What are the dependency injection types?

There are three common dependency injection types in Spring: constructor injection, setter injection, and field injection.

Constructor injection is the recommended way. It makes required dependencies explicit when the bean is created. It also helps prevent null pointer exceptions because Spring has to provide all required dependencies during application startup.

Constructor injection is also easier for unit testing because we can pass mock objects directly.

Setter injection is useful for optional dependencies.

Field injection is simple, but it is not recommended because it hides dependencies and makes testing harder.

A related issue is circular dependency. For example, bean A depends on bean B, and bean B also depends on bean A. The better solution is to refactor the design. If we really need to break the initialization cycle, we can use `@Lazy`.



# How do you implement exception handling in Spring Boot?

I would answer this from two layers: exception definition and exception handling.

First, for exception definition, it goes back to Java exceptions. Java has checked exceptions and unchecked exceptions. In Spring Boot business logic, we usually define custom runtime exceptions for business errors, such as resource not found, invalid request, or duplicate operation.

Second, for exception handling, we usually do not handle every exception inside each controller method. Instead, we use a global exception handler.

The common way is to use `@RestControllerAdvice` with `@ExceptionHandler`. This allows us to catch exceptions globally and return a consistent response body and HTTP status code.

Another style is AOP. We can use `@Aspect`, `@Pointcut`, and advice like `@Around`, `@AfterThrowing`, `@Before`, `@After`, and `@AfterReturning` to handle cross-cutting logic, especially exception logging or monitoring.

So my structure is: define custom exceptions first, then handle them globally, and finally return a consistent API response to the client.



# CompletableFuture / multithreading in Spring Framework

CompletableFuture is a Java 8 feature for asynchronous programming.

Compared with old Future, CompletableFuture provides better APIs for chaining, combining, and handling async results. We can use it when we want to run several independent tasks in parallel and combine the results later.

In a Spring Boot application, I usually use it in the service layer. For example, if one request needs to call multiple independent services or process multiple chunks of data, we can start multiple CompletableFuture tasks, then use allOf and join to combine the results.

But in Spring, we should not manually create many threads. A better way is to configure a thread pool and submit async tasks to that thread pool.

Spring also provides `@Async`. To use it, we need to enable async support with `@EnableAsync`, and then configure an executor. If we have multiple thread pools, we can use a name or qualifier to decide which pool should execute which task.

In real projects, we may split thread pools by task type. For example, order processing can use a core thread pool, while email notification or report generation can use an auxiliary thread pool. This prevents less important tasks from blocking core business tasks.

For thread safety, we also need to consider Java concurrency tools, such as AtomicInteger, AtomicReference, ConcurrentHashMap, synchronized, Lock, and ReentrantLock.



# What annotations do we use to configure customized Actuator?

For most projects, we usually do not customize Actuator too much. We normally expose existing Actuator endpoints like health, metrics, and prometheus, then connect them with Prometheus and Grafana.

If we really need to create a custom Actuator endpoint, we can use `@Endpoint`.

Inside the endpoint, we can use `@ReadOperation` for read-only operations, `@WriteOperation` for write operations, and `@DeleteOperation` for delete operations.

The idea is that Actuator provides production-ready monitoring endpoints. Custom Actuator is more common for platform teams or logging/monitoring teams. For normal business service teams, we usually just configure and expose existing endpoints.



# How does Spring IOC work?

Spring IOC means Inversion of Control. In Spring, object creation and dependency management are controlled by the Spring IOC container.

When the application starts, Spring creates the ApplicationContext, which is the IOC container.

Then Spring Boot uses `@SpringBootApplication`. This annotation includes auto configuration, configuration registration, and component scanning.

During component scanning, Spring scans annotations like `@Controller`, `@RestController`, `@Service`, `@Repository`, and `@Component`. These classes are registered as Spring beans.

For third-party classes or classes we cannot annotate directly, we can use `@Configuration` and `@Bean` to register them into the container.

After beans are registered, Spring creates bean instances and manages their lifecycle. The default bean scope is singleton. Other scopes include prototype, request, session, and application.

Then Spring performs dependency injection. The common injection types are constructor injection, setter injection, and field injection. Constructor injection is recommended because it makes required dependencies clear, helps prevent null pointer exceptions, and is easier for testing.

If there are multiple beans of the same type, Spring may not know which one to inject. In that case, we can use `@Qualifier` to specify the exact bean.

So the whole flow is: scan classes, register bean definitions, create beans, manage bean scopes and lifecycle, and inject dependencies.



# Could AOP apply to a private method?

No, normal Spring AOP cannot apply to a private method.

Spring AOP is proxy-based. The method call has to go through the Spring proxy. A private method cannot be overridden by the proxy, and it cannot be called from outside the class through the proxy.

Also, even if the method is public, if it is called internally from another method in the same class, it may still bypass the proxy. This is called self-invocation.

There are two common solutions.

First, move the method to another Spring bean and call it from outside. Then the call goes through the Spring proxy, so AOP can intercept it.

Second, call the method through the Spring proxy, for example by self-injecting the proxied bean or using the current proxy. But this is less clean, so I prefer extracting the logic to another bean.

If we really need to intercept private methods, we may need AspectJ weaving, but that is not the common Spring Boot AOP style.
