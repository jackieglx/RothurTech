# jdbc vs hibernate

JDBC and Hibernate are both technologies used by Java applications to access relational databases.

In terms of their relationship, Hibernate usually communicates with the database through JDBC, so Hibernate can be considered a higher-level abstraction built on top of JDBC.

In terms of abstraction, JDBC is a low-level database access API, while Hibernate is a high-level ORM framework.

In terms of SQL, JDBC usually requires developers to write SQL manually, while Hibernate can generate SQL based on entity mappings.

In terms of object mapping, JDBC requires developers to manually convert a `ResultSet` into Java objects, while Hibernate automatically maps database tables to Java objects.

In terms of use cases, JDBC is more suitable when we need direct control over SQL and performance, while Hibernate is more suitable for applications with complex entity relationships and many CRUD operations.



JDBC 和 Hibernate 都是 Java 应用访问关系型数据库的技术。

在二者关系方面，Hibernate 底层通常仍然通过 JDBC 与数据库通信，因此 Hibernate 可以看作是对 JDBC 的更高层封装。

在抽象层面，JDBC 是底层数据库访问 API，而 Hibernate 是高层 ORM 框架。

在 SQL 方面，JDBC 通常需要开发者手动编写 SQL，而 Hibernate 可以根据实体映射自动生成 SQL。

在对象映射方面，JDBC 需要手动将 `ResultSet` 转换成 Java 对象，而 Hibernate 可以自动完成对象与数据库表之间的映射。

在适用场景方面，JDBC 更适合需要直接控制 SQL 和性能的场景，而 Hibernate 更适合实体关系复杂、CRUD 操作较多的业务系统。



# Statement vs PreparedStatement vs CallableStatement

`Statement`, `PreparedStatement`, and `CallableStatement` are JDBC interfaces used to execute SQL statements against a database.

In terms of SQL syntax, `Statement` executes a complete SQL string, while `PreparedStatement` uses `?` placeholders to bind parameters.

In terms of security, `Statement` often relies on string concatenation and may cause SQL Injection, while `PreparedStatement` can effectively prevent SQL Injection.

In terms of performance, `PreparedStatement` can precompile and reuse the same SQL statement, so it is usually more efficient than repeatedly using `Statement`.

In terms of functionality, `CallableStatement` is mainly used to call Stored Procedures and supports both input and output parameters.

In terms of use cases, `Statement` is suitable for simple static SQL, `PreparedStatement` is suitable for parameterized queries and updates, and `CallableStatement` is suitable for calling Stored Procedures.



`Statement`、`PreparedStatement` 和 `CallableStatement` 都是 JDBC 中用于向数据库执行 SQL 的接口。

在 SQL 编写方面，`Statement` 直接执行完整的 SQL 字符串，而 `PreparedStatement` 使用 `?` 占位符绑定参数。

在安全性方面，`Statement` 通过字符串拼接传入数据，容易产生 SQL Injection，而 `PreparedStatement` 可以有效防止 SQL Injection。

在性能方面，`PreparedStatement` 可以预编译并重复执行同一条 SQL，通常比重复使用 `Statement` 更高效。

在功能方面，`CallableStatement` 主要用于调用数据库中的 Stored Procedure，并且支持输入参数和输出参数。

在适用场景方面，`Statement` 适合简单的静态 SQL，`PreparedStatement` 适合带参数的普通查询和更新，而 `CallableStatement` 适合调用 Stored Procedure。



# how to prevent sql injection

The most important way to prevent SQL injection is to use parameterized queries, such as `PreparedStatement` in JDBC.

Do not directly concatenate user input into an SQL string, because attackers may use malicious input to change the query logic.

We should also validate the format, length, and type of user input, but input validation cannot replace parameterized queries.

The database account should follow the principle of least privilege and should not have unnecessary permissions, such as deleting databases or changing table structures.

In Hibernate, JPA, or Spring Data JPA, we should also use parameter binding instead of building queries through string concatenation.



防止 SQL Injection 最重要的方法是使用参数化查询，例如 JDBC 中的 `PreparedStatement`。

不要把用户输入直接拼接到 SQL 字符串中，因为攻击者可能输入恶意 SQL 改变原来的查询逻辑。

同时应该对用户输入进行格式、长度和类型校验，但输入校验不能替代参数化查询。

数据库账号应该遵循最小权限原则，避免应用账号拥有删除数据库或修改表结构等不必要的权限。

在 Hibernate、JPA 或 Spring Data JPA 中，也应该使用参数绑定，而不是通过字符串拼接动态生成查询。





# what is ORM

ORM stands for  Object-Relational Mapping. It is a technology that maps Java objects to relational database tables.

It usually maps a Java class to a table and maps the object's fields to the table's columns.

With ORM, developers can work with Java objects without manually writing a large amount of SQL for every CRUD operation.

An ORM framework can generate SQL and convert database query results into Java objects.

It can reduce repetitive code and improve development efficiency, but it may generate inefficient SQL if it is not used properly.

Hibernate is a common ORM framework, while JPA is an ORM standard defined by Java.



ORM（Object-Relational Mapping）是一种把 Java 对象和关系型数据库表进行映射的技术。

它通常把一个类映射为一张表，把对象属性映射为表中的列。

通过 ORM，开发者可以操作 Java 对象，而不需要为每个 CRUD 操作手动编写大量 SQL。

ORM 框架会负责生成 SQL，并把数据库查询结果转换成 Java 对象。

它可以减少重复代码、提高开发效率，但使用不当时也可能生成低效 SQL。

Hibernate 是常见的 ORM 框架，而 JPA 是 Java 定义的一套 ORM 标准。



# JPA vs Hibernate

JPA is an official ORM specification defined by Java. It defines standards for entity mapping, persistence operations, and query APIs.

Hibernate is a specific ORM framework and one of the most common implementations of JPA.

JPA cannot perform database operations by itself. It must rely on a JPA provider such as Hibernate.



JPA 是 Java 官方定义的 ORM 规范，规定了实体映射、持久化操作和查询接口

Hibernate 是一个具体的 ORM 框架，也是 JPA 最常见的实现之一。

JPA 本身不能独立完成数据库操作，必须依赖 Hibernate 等 JPA Provider



# what are the Persistent State in Entity LifeCycle

Persistent State means that an entity object is associated with the JPA Persistence Context and is managed by the EntityManager.

In this state, the entity usually represents a record in the database, or it is waiting to be inserted after `persist()` is called.

When we modify the fields of a persistent entity, we usually do not need to call an update method manually.

JPA uses Dirty Checking to detect the changes automatically.

The changes are synchronized with the database when the transaction is flushed or committed.

Persistent State is also commonly called Managed State, and an entity returned by `find()` is usually in this state.



**Persistent State 就是实体对象已经和 JPA 的 `Persistence Context` 建立关联，并且正在被 `EntityManager` 管理的状态。**

在这个状态下，实体通常已经对应数据库中的一条记录，或者执行 `persist()` 后即将被插入数据库。
 当你修改 Persistent Entity 的字段时，通常不需要再次调用 `update()`，JPA 会通过 **Dirty Checking** 检测变化。
 在事务 `flush` 或提交时，这些修改会自动同步到数据库。
 Persistent State 也经常被称为 **Managed State**，两者通常表示同一个状态。
 例如，从数据库中通过 `find()` 查询出来的实体，默认就是 Persistent State。





# Mapping relationship

In JPA, mapping relationships are used to represent associations between entity classes and map those relationships to database tables.

Common relationship types include one-to-one with `@OneToOne`, one-to-many with `@OneToMany`, many-to-one with `@ManyToOne`, and many-to-many with `@ManyToMany`.



JPA 中的 Mapping Relationship 用来表示实体类之间的关联关系，并把这种关系映射到数据库表。

常见关系包括一对一 `@OneToOne`、一对多 `@OneToMany`、多对一 `@ManyToOne` 和多对多 `@ManyToMany`。

# what is the cascade type

In JPA, Cascade Type controls whether an operation on one entity is automatically passed to its related entities.

`CascadeType.PERSIST` means that when the parent entity is saved, the related entities are also saved automatically.

`CascadeType.MERGE` means that when the parent entity is updated or merged, the related entities are also merged.

`CascadeType.REMOVE` means that when the parent entity is deleted, the related entities are also deleted, so it should be used carefully.

`CascadeType.REFRESH` and `CascadeType.DETACH` pass refresh and detach operations to the related entities.

`CascadeType.ALL` includes all cascade types.



JPA 中的 Cascade Type 用来控制对一个实体执行操作时，是否把相同操作自动传递给关联实体。

`CascadeType.PERSIST` 表示保存父实体时，也自动保存关联实体。

`CascadeType.MERGE` 表示更新或合并父实体时，也合并关联实体。

`CascadeType.REMOVE` 表示删除父实体时，也删除关联实体，因此使用时要特别小心。

`REFRESH`、`DETACH` 分别传播刷新和分离操作，而 `ALL` 表示包含所有 Cascade Type。



# what is the fetch type

In JPA, Fetch Type controls whether related entities are loaded immediately or loaded later.

`FetchType.EAGER` means the related entity is loaded immediately when the main entity is loaded.

`FetchType.LAZY` means the related entity is not loaded at first. It is loaded only when we access it.

By default, `@ManyToOne` and `@OneToOne` use EAGER, while `@OneToMany` and `@ManyToMany` use LAZY.

In real projects, LAZY is usually preferred because EAGER may load a large amount of unnecessary data.

When using LAZY, we should be careful about the N+1 Query problem and `LazyInitializationException`. We can use `JOIN FETCH` or Entity Graph to solve these problems.



JPA 中的 Fetch Type 用来控制查询一个实体时，关联实体是立即加载还是延迟加载。

`FetchType.EAGER` 表示加载主实体时，立即加载关联实体。

`FetchType.LAZY` 表示先不查询关联实体，只有真正访问它时才进行加载。

JPA 默认情况下，`@ManyToOne` 和 `@OneToOne` 使用 EAGER，而 `@OneToMany` 和 `@ManyToMany` 使用 LAZY。

实际项目通常更推荐 LAZY，因为 EAGER 可能查询大量不需要的数据。

使用 LAZY 时要注意 N+1 Query 和 `LazyInitializationException`，可以使用 `JOIN FETCH` 或 Entity Graph 解决。



# what is the first/second level cache

First-Level Cache and Second-Level Cache are both used to reduce repeated database queries and improve ORM query performance.

In terms of scope, First-Level Cache belongs to a single `EntityManager` or Hibernate `Session`, while Second-Level Cache belongs to the entire `EntityManagerFactory` or `SessionFactory`.

In terms of sharing, First-Level Cache cannot be shared across different sessions, while Second-Level Cache can be shared by multiple sessions.

In terms of configuration, First-Level Cache is enabled by default, while Second-Level Cache usually needs to be configured manually.

In terms of lifecycle, First-Level Cache is cleared when the session is closed, while Second-Level Cache usually has a longer lifecycle.

In terms of use cases, First-Level Cache is suitable for avoiding repeated queries within one transaction, while Second-Level Cache is suitable for data that is read frequently but updated less often.



First-Level Cache 和 Second-Level Cache 都用于减少重复的数据库查询，提高 ORM 的查询性能。
 在作用范围方面，First-Level Cache 属于单个 `EntityManager` 或 Hibernate `Session`，Second-Level Cache 属于整个 `EntityManagerFactory` 或 `SessionFactory`。
 在共享范围方面，First-Level Cache 不能被不同 Session 共享，而 Second-Level Cache 可以被多个 Session 共享。
 在启用方式方面，First-Level Cache 默认自动开启，Second-Level Cache 通常需要手动配置缓存实现。
 在生命周期方面，First-Level Cache 会随着 Session 关闭而失效，Second-Level Cache 的生命周期通常更长。
 在适用场景方面，First-Level Cache 适合避免单个事务中的重复查询，Second-Level Cache 适合缓存经常读取但很少修改的数据。

# left join vs right join vs inner join vs outer join vs cross join

These joins are all used to combine data from two or more database tables based on certain rules.

`INNER JOIN` returns only the rows that match the join condition in both tables.

`LEFT JOIN` returns all rows from the left table. If there is no matching row in the right table, the right-side columns will be `NULL`.

`RIGHT JOIN` returns all rows from the right table. If there is no matching row in the left table, the left-side columns will be `NULL`.

`FULL OUTER JOIN` returns all rows from both tables. If a row has no match on the other side, the missing columns will be `NULL`.

`CROSS JOIN` does not require a join condition. It returns the Cartesian product of the two tables, so the total number of rows is the number of rows in the first table multiplied by the number of rows in the second table.



这些 Join 都用于根据一定规则组合两个或多个数据库表中的数据。

`INNER JOIN` 只返回左右两张表中能够满足连接条件的记录。

`LEFT JOIN` 返回左表的所有记录，没有匹配的右表字段会显示为 `NULL`。

`RIGHT JOIN` 返回右表的所有记录，没有匹配的左表字段会显示为 `NULL`。

`FULL OUTER JOIN` 返回两张表的所有记录，任何一侧没有匹配的字段都会显示为 `NULL`。

`CROSS JOIN` 不需要连接条件，它会返回两张表的笛卡尔积，因此结果数量等于两张表记录数的乘积。



# union vs union all

`UNION` and `UNION ALL` are both used to combine the results of multiple `SELECT` queries into one result set.

In terms of duplicate data, `UNION` removes duplicate rows, while `UNION ALL` keeps all duplicate rows.

In terms of performance, `UNION` is usually slower because it needs an extra deduplication step.

In terms of the number of rows, `UNION` may return fewer rows, while `UNION ALL` returns all rows from all queries.

In terms of requirements, both of them require the same number of columns, and the matching columns must have compatible data types.

In terms of use cases, use `UNION` when you need to remove duplicates, and use `UNION ALL` when duplicates are allowed and better performance is preferred.



`UNION` 和 `UNION ALL` 都用于把多个 `SELECT` 查询的结果纵向合并成一个结果集。
 在重复数据方面，`UNION` 会自动删除重复记录，而 `UNION ALL` 会保留所有重复记录。
 在性能方面，`UNION` 需要额外进行排序或去重，因此通常比 `UNION ALL` 更慢。
 在结果数量方面，`UNION` 返回的记录数可能减少，而 `UNION ALL` 返回所有查询结果的总记录数。
 在使用要求方面，两者都要求每个 `SELECT` 返回相同数量的列，并且对应列的数据类型需要兼容。
 在适用场景方面，需要去重时使用 `UNION`，不需要去重时通常优先使用性能更好的 `UNION ALL`。
